## 串讲词（约 60 秒）

> 我来解释从 ELF 文件到运行中进程的完整过程。
>
> **先看左图——ELF 文件在磁盘上的结构。** 文件头部是 52 字节的 ELF Header，包含四个关键信息：magic number 用来校验这是合法的 ELF 文件，e_entry 是程序的入口地址，e_phoff 指向 Program Header 表的位置，e_phnum 告诉我们有几个段。这张图里有两个段——一个是 text 代码段，另一个是 data 数据段。注意第二个段的 memsz 比 filesz 多了 0x2000 字节——这多出来的就是 BSS 段，对应 C 语言里未初始化的全局变量，它们在文件中不占空间，但运行时需要占用内存。
>
> **中间这张图是我实现的加载流程。** 入口是 naive_uload 函数。第一步，write_user_args 把命令行参数 argc 和 argv 写入固定地址 0x8FFF0000——这是我设计的参数传递机制。第二步，loader 函数打开 ELF 文件，读 Header，遍历 Program Header 表。对每个标记为 PT_LOAD 的段，先 seek 到文件偏移位置，再 read 到目标虚拟地址，如果 memsz 大于 filesz，就用 memset 把差值部分清零——这就是 C 语言全局变量初始化为零的底层实现。最后返回 e_entry 入口地址。第三步，直接跳转——CPU 开始执行用户程序。
>
> **右图是程序运行时的内存布局。** 低地址放代码段 text 和数据段 data，BSS 紧接其后——注意数据和 BSS 段的内容都是从 ELF Header 的 Program Header 表里读取的。BSS 的上方是 heap，由 brk 系统调用管理——我实现的 loader 会自动把 brk 初始化为 BSS 结束位置页对齐后的值。再往上是用户栈，从高地址向下增长。最顶上那块是参数区——0x8FFF0000 存放着 argc 和 argv，crt0 启动代码从这里读取参数，重建 argv 数组，然后调用 main 函数。
>
> 这三张图串起来就是：**磁盘上的 ELF 文件 → 加载器解析并拷贝到内存 → 运行时进程地址空间**。整个流程由我实现的 loader.c 中的 naive_uload 函数完成。

## 如果老师追问

**Q: memsz 为什么可以大于 filesz？**

> 这是编译器的优化。未初始化的全局变量不需要在文件中存储实际数据，只需要在加载时分配零填充的内存。所以 filesz 只包含有初始值的数据，memsz 包含全部内存需求。差值就是 BSS 段的大小。我实现的 loader 用 memset 把这个差值区域全部填零，保证 C 语言的语义正确。

**Q: 为什么要页对齐 brk？**

> 内存管理以页（4KB）为单位。如果 brk 不对齐，后续的 malloc 实现会很复杂。页对齐后，堆从完整的页面边界开始，简化了内存分配器的实现。
