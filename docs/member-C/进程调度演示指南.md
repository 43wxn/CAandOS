# 进程调度器现场演示指南

## 做了什么修改

| 修改 | 效果 |
|------|------|
| dterm PCB 移入 `pcb[2]` | 调度器能找到 shell 并切回来 |
| 修复调度器同进程跳过逻辑 | 不会陷入"找到自己就返回"的死循环 |
| demo 线程产生可见 Log 输出 | 能亲眼看到每个线程在运行 |
| demo 线程运行后自动退出 | 退出后调度器自动切回 shell |

## 现场演示步骤（20 秒）

### 步骤 1：查看初始状态

```
demo
```

在 NEMU **终端**（不是 NEMU 图形窗口）中会看到类似输出：

```
[demo] created 3 threads (logger=10 worker=11 watchdog=12)
[demo] yielding to start Round-Robin scheduling...
[logger] tick=0  <-- running on CPU
[worker] tick=0  <-- running on CPU
[watchdog] tick=0  <-- running on CPU
[logger] tick=1  <-- running on CPU
[worker] tick=1  <-- running on CPU
[watchdog] tick=1  <-- running on CPU
[logger] tick=2  <-- running on CPU
[logger] finished, exiting
[worker] tick=2  <-- running on CPU
[worker] finished, exiting
[watchdog] tick=2  <-- running on CPU
[watchdog] finished, exiting
[demo] all threads finished, back to shell
```

> 注意：输出在 NEMU 的终端（运行 `make run` 的那个窗口），不在 NEMU 图形窗口！

### 步骤 2：验证 shell 正常

Shell 仍然正常工作（命令历史保留，登录状态保留）。

## 答辩口播词

> "现在我演示进程调度器。输入 demo 命令，内核创建了三个线程——logger、worker、watchdog。接下来可以看到 Round-Robin 调度器在三个线程之间轮转——每个线程打印自己的名字和计数后主动 yield 让出 CPU，调度器选择下一个就绪线程。三轮之后所有线程自动退出，shell 恢复正常。
>
> 这证明了：1）调度器能正确选择下一个就绪进程；2）上下文切换在 trap.S 中正确完成；3）进程退出后资源可回收，系统继续正常运行。"

## 如果老师问"怎么证明是调度器在工作？"

> "关键证据是输出**交替**出现。如果只有一个进程在跑，输出会是一个线程连续打印三次。但实际上 logger→worker→watchdog→logger→worker→watchdog 交替出现，这证明了调度器在三个线程之间轮转。每个线程的 yield 系统调用触发调度，调度器找到下一个 RUNNABLE 进程并切换。"

## 代码走读（如果老师深入提问）

切到 `nanos-lite/src/proc.c`，展示三个关键函数：

1. **`proc_create_thread()`**（~195 行）— 手动初始化 Context：mepc 指向线程入口函数，mstatus=0x1880，独立 16KB 栈
2. **`proc_schedule()`**（~225 行）— Round-Robin：遍历 PCB 数组，跳过当前进程，找第一个 RUNNABLE
3. **`demo_thread()`**（~258 行）— 打印 Log → yield → 重复 3 次 → 标记 ZOMBIE 退出

## 故障应急

| 问题 | 处理 |
|------|------|
| 输出没有出现在终端 | 检查是否在看正确的窗口（终端窗口，非图形窗口） |
| Shell 卡住不动 | 等 5 秒，线程会自动退出。如果还卡住，关掉 NEMU 重启 |
| demo 命令不存在 | 重新编译：`make -C nanos-lite ARCH=riscv32-nemu update` |
