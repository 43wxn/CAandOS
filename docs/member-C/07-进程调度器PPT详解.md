### 架构图怎么画（上半部分）

**用 PPT 的表格工具：**

1. 插入 → 表格 → 4 列 × 2 行
2. 每个格子是一个 PCB 槽位，填入内容：

| 格子 0 | 格子 1 | 格子 2 | 格子 3 |
|--------|--------|--------|--------|
| `[0] idle` | `[1] init` | `[2] dterm` | `[3] logger` |
| RUNNING 灰 | RUNNING 灰 | **RUNNABLE 绿** | **RUNNABLE 绿** |

| 格子 4 | 格子 5 | 格子 6 | 格子 7 |
|--------|--------|--------|--------|
| `[4] worker` | `[5] watchdog` | `[6] —` | `[7] —` |
| **RUNNABLE 绿** | **RUNNABLE 绿** | UNUSED 暗 | UNUSED 暗 |

3. 绿色格子 = RUNNABLE（可被调度），灰色 = RUNNING（不会被调度），暗色 = UNUSED
4. 在表格右侧画一个大箭头 → 标注 "last_idx 遍历方向"
5. 表格下方放 4 行伪代码（用等宽字体 Consolas 14pt）

### 口播词（约 40 秒）

> 这页展示进程调度器的架构和现场效果。上半部分是 8 槽位 PCB 数组——槽位 0 和 1 是 idle 和 init，始终 RUNNING 不参与调度。槽位 2 是 dterm shell，槽位 3 到 5 是三个 demo 线程。绿色表示 RUNNABLE——可被调度。
>
> 调度算法只有 15 行 C 代码：保存当前进程上下文，从上次位置开始遍历数组，跳过当前进程，找到第一个 RUNNABLE 的进程，返回其 Context 指针。
>
> 下半部分是现场 demo 效果。输入 demo 命令后，三个线程的日志交替出现——这直接证明了 Round-Robin 在正确轮转。按 ESC 停止，shell 恢复正常。这是最有说服力的现场演示。

## 第 2 页：上下文切换（汇编级）

### 页面标题：RISC-V 上下文切换 — trap.S 汇编分析

### 页面布局（左右结构）

```
┌────────────────────────────┬──────────────────────────┐
│  左栏 (55%): trap.S 代码    │  右栏 (45%): 图解          │
│                            │                          │
│  __am_asm_trap:            │  Context 结构体 (144B):   │
│    addi sp, sp, -144       │  ┌─────────────────┐    │
│                            │  │ gpr[0]  (x0)    │ +0 │
│    sw x1,  4(sp)           │  │ gpr[1]  (ra)    │ +4 │
│    sw x3, 12(sp)           │  │ gpr[2]  (sp)    │ +8 │
│    ...                     │  │ gpr[3]  (gp)    │+12 │
│    sw x31, 124(sp)         │  │ ...             │    │
│                            │  │ gpr[31] (t6)    │+124│
│    csrr t0, mcause         │  ├─────────────────┤    │
│    csrr t1, mstatus        │  │ mcause          │+128│
│    csrr t2, mepc           │  │ mstatus         │+132│
│                            │  │ mepc            │+136│ ← ★
│    mv a0, sp      ← ①     │  │ pdir            │+140│
│    call __am_irq_handle    │  └─────────────────┘    │
│    mv sp, a0      ← ②     │                          │
│                            │  ① mv a0,sp: 传指针     │
│    LOAD t1, 132(sp)        │  ② mv sp,a0: 收指针     │
│    LOAD t2, 136(sp)        │  ③ mret:     跳转      │
│    csrw mstatus, t1        │                          │
│    csrw mepc, t2           │  如果 a0 返回的是另一个    │
│                            │  进程的 Context*，mret    │
│    lw x1, 4(sp)            │  就完成了进程切换!        │
│    ...                     │                          │
│    LOAD sp, 8(sp)          │                          │
│    mret           ← ③     │                          │
└────────────────────────────┴──────────────────────────┘
```

### 左栏怎么画

1. 深色背景矩形 (#0D1117)，白色等宽文字 Consolas 13pt
2. **三行关键代码用青色高亮**（`mv a0, sp`、`mv sp, a0`、`mret`）
3. 这三行旁边标注 ① ② ③ 圆圈
4. 其他代码保持白色，不需要高亮

### 右栏怎么画

1. 上方：用 PPT 矩形工具画 Context 结构体布局图
   - 灰色矩形 + 白色文字标注偏移量
   - mepc 那一行用**黄色背景**高亮，标注 "★ mret 跳转目标"
2. 下方：三行文字解释 ①②③
   - ① 传 Context 指针给 C 调度器
   - ② 接收调度器返回的新 Context 指针
   - ③ 跳转到目标进程

### 口播词（约 40 秒）

> 这页深入到汇编级别。trap.S 是整个操作系统最底层的 86 行汇编代码。每次 ecall 陷入内核时，CPU 执行的第一条指令在这里。
>
> 左栏是核心代码。先分配 144 字节栈空间——这就是 Context 结构体的大小。然后保存 32 个通用寄存器和 3 个 CSR——mcause 异常原因、mstatus 机器状态、mepc 返回地址。关键在中间这三步。
>
> 第一步，mv a0,sp——把当前进程的 Context 指针传给 C 语言写的调度器。调度器遍历 PCB 数组，选中下一个进程，返回那个进程的 Context 指针。
>
> 第二步，mv sp,a0——这是进程切换的关键！如果调度器返回的是另一个进程的 Context 指针，sp 现在就指向那个进程的寄存器快照。
>
> 第三步，从新 Context 恢复 mepc、恢复 32 个寄存器、恢复栈指针，然后 mret。mret 指令把 mepc 的值写入 PC——如果这个 mepc 是另一个进程保存的，CPU 就跳转到了那个进程——进程切换完成。
>
> 整个切换不需要任何魔法，就是保存 A 的寄存器 → 加载 B 的寄存器 → mret 跳转。

## 答辩追问准备

**Q: 为什么只有 8 个槽位？**

> 静态数组，编译时确定。嵌入式 OS 场景下 8 个足够。可以改为动态分配，但当前阶段优先保证稳定性。

**Q: 调度器是抢占式还是协作式？**

> 当前是协作式——进程通过 yield() 系统调用主动让出 CPU。抢占式调度的框架已预留（TIME_SHARING 宏），启用后时钟中断会触发强制调度。

**Q: 如果所有进程都在等待，调度器怎么办？**

> 调度器返回 prev——继续当前进程。当前进程至少是 dterm shell，它会在事件循环中等待键盘输入。所以系统永远不会完全卡死。

**Q: proc_schedule 的时间复杂度？**

> O(n)，n = MAX_NR_PROC = 8。对于嵌入式场景完全足够。可以优化为 O(1) 用链表，但当前数组实现更简单、更安全。
